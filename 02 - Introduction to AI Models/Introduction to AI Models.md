# Introduction to AI Models

## What are LLM ?

A **large language model (LLM)** is a system trained to understand and generate human language. It is called **“large”** because it is trained on massive amounts of text, including books, articles, forums, code, documentation, and more, and because it contains **billions of parameters**. During training, it learns statistical patterns in language such as grammar, sentence structure, tone, common facts, and phrasing.

When I give an LLM a prompt, it does not look up the answer like a search engine. It predicts what a likely and useful response should look like based on patterns it has seen before. In practice, it is a giant mathematical structure that works through probability. It does not truly understand language the way humans do, it does not have beliefs, and it does not have intelligence in the human sense. That is why it can produce answers that sound fluent and convincing without necessarily being true.

### Why Output Quality Can Vary

The quality of the model depends heavily on its training data. If the data is biased, inaccurate, or low quality, the outputs will reflect that. This is especially important for **code generation**. These models are trained on huge amounts of public code, including code that is outdated, poorly written, insecure, or full of anti-patterns. As a result, generated code can look clean and professional while still being buggy, insecure, or hard to maintain.

### What Matters for Developers

Training a model from scratch requires enormous amounts of data, GPUs, and infrastructure, so that is not something most developers will do themselves. As developers, the real goal is to understand how to use these models: how to prompt them effectively, how to handle their limitations, and how to integrate them into applications to build smarter features.


## What can you do with LLM?

When integrating a language model into an application, it is usually **not the center of the system**. Instead, it acts as a **supporting component**. A typical architecture includes a **front end, a backend, a database**, and then the **model**, which receives input and returns output that improves the user experience or automates part of a feature.

### Common Use Cases

Language models can be used for many practical tasks within applications, including:

* **Summarization** – condensing long content such as reviews, documents, or reports into shorter summaries
* **Content generation** – creating emails, product descriptions, marketing text, or documentation
* **Text classification** – determining whether something is spam, identifying sentiment, or categorizing support tickets
* **Translation** – converting text between different languages
* **Information extraction** – pulling structured data from messy sources such as PDFs, forms, or long text
* **Chatbots and Q&A systems** – answering user questions based on documentation, company knowledge bases, or user data

For example, an application might summarize product reviews, generate email responses, categorize incoming support requests, translate user messages, extract data from documents, or answer questions using internal business documents.

### The Basic Interaction Pattern

In most applications, the interaction with a language model follows a simple pattern:

1. **Send a prompt** to the model.
2. **Receive a response** generated by the model.

The response can be returned in many formats depending on what the application needs, including:

* Plain text
* JSON
* Arrays
* Numbers
* Images
* Other structured formats

### The Practical Developer Mindset

The key idea is that developers usually **do not build the model themselves**. Training large models requires enormous amounts of data, GPUs, and infrastructure. Instead, the focus is on **using existing models effectively**.

For developers, the real challenge is learning how to **integrate language models into real products**: designing prompts, handling their limitations, structuring outputs, and connecting the model to the rest of the application stack.



## Understanding Tokens and Context Window

When a prompt is sent to a language model, the model does not process the input as plain text. Instead, it first breaks the text into smaller units called **tokens**. A token can represent a whole word, part of a word, punctuation, a space, or even an emoji. Because of this, tokens are not the same as characters or words—they sit somewhere in between.

### Why Tokens Matter

Tokens are important for two main reasons: **cost** and **technical limits**.

**Cost** is often calculated based on the number of tokens used, particularly the number of **output tokens** generated by the model. If an application produces large amounts of text or processes long documents, the cost can increase quickly. For that reason, choosing a model should not simply mean selecting the newest or most powerful one. The best choice is the model that fits the actual needs of the application.

### Context Window Limits

Another key concept is the **context window**. This is the maximum number of tokens a model can process in a single request. The context window includes:

* The **prompt**
* The **model’s response**
* Any **conversation history** when building chat-based experiences

If the total number of tokens exceeds the model’s context window, the model may stop generating text or the output may be cut off.

### Practical Takeaway

Understanding tokens helps developers:

* Estimate **cost**
* Stay within **model context limits**
* Choose the **right model for a specific use case**

This makes token awareness an important part of designing applications that integrate language models.


### Counting tokens in code with TikToken

Now I want to show how to count tokens programmatically using code. I start by creating a small Node.js playground project where I can experiment locally.

First, I open a terminal, move to a location like the Desktop, create a new folder called `playground`, enter it, and initialize a Node project.

```bash
mkdir playground
cd playground
npm init -y
```

This creates a `package.json` file without asking interactive setup questions.

Next, I install `tiktoken`, which is the tokenizer library used for OpenAI models. Different AI providers may use different tokenizers, so this is specific to OpenAI-style token counting.

```bash
npm install tiktoken
```

Then I open the project in VS Code.

```bash
code .
```

If that command does not work, I can just drag the folder into VS Code manually.

#### Create the script

Now I add a file called `index.js`.

At the top, I import `getEncoding` from `tiktoken`. Then I create an encoding object using `cl100k_base`, which is a common encoding used by many OpenAI models. An encoding is basically a dictionary that maps token IDs to actual token pieces.

After that, I call `encoding.encode()` with a piece of text. This returns an array of token IDs. Finally, I print that array to the console.

```js
import pkg from "tiktoken";
const { get_encoding } = pkg;

const encoding = get_encoding("cl100k_base");

const text = "Hello world, this is the first test of the TikToken library.";
const tokens = encoding.encode(text);

console.log(tokens);
console.log("Token count:", tokens.length);
```

> Why `cl100k_base`? `cl100k_base` is the encoding name. You can think of it as the tokenizer vocabulary. The `100k` part means the encoding contains roughly 100,000 token entries. Each token ID maps to some text fragment, which could be a full word, part of a word, punctuation, or whitespace.

#### Run the script

Now I open the integrated terminal in VS Code and run the file with Node.js.

```bash
node index.js
```

At first, I may get this error:

```txt
SyntaxError: Cannot use import statement outside a module
```

This happens because Node.js treats JavaScript files as CommonJS by default, while `import` uses ES module syntax.

#### Fix the module error

To tell Node.js to use modern ES modules, I open `package.json` and add `"type": "module"`.

```json
{
  "name": "playground",
  "version": "1.0.0",
  "type": "module",
  "main": "index.js"
}
```

After that, I rerun the script:

```bash
node index.js
```

Now it works, and I get an array of numbers. Each number is a token ID, and the total number of items in the array is the number of tokens in the input text.

#### What the output means

The output looks something like this:

```js
[9906, 1917, 11, 420, 374, 279, 1176, 1296, 315, 279, 350, 1609, 5390, 13]
```

Each number represents a token. These are not human-readable words by themselves. They are internal IDs that correspond to token pieces in the encoding vocabulary.

What matters most in practice is usually the token count:

```txt
Token count: 13
```

#### Why this is useful

When working with long prompts, documents, or chat history, I can use `tiktoken` to count tokens before sending anything to a language model. This helps me:

* estimate cost
* stay within the model’s context window
* decide whether I need to truncate, split, or compress input text

So the practical takeaway is simple: if I want to control cost and avoid token limit errors, I should count tokens before making LLM requests.


## [Choosing the Right AI Model](https://developers.openai.com/api/docs/models/compare)

There are hundreds of AI models available, and new ones appear constantly. Because of this, there is **no single best model for every situation**. The right choice depends on the application and its specific requirements. Instead of focusing on model names—which change quickly—it is more useful to focus on the **criteria that actually matter**.

### 1. Reasoning Ability

The first question to ask is **how intelligent the model really needs to be**.

If the task involves solving complex problems, planning, or deep reasoning, a more advanced model with stronger reasoning capabilities is necessary. However, many tasks do not require such powerful models.

For simpler tasks such as:

* Extracting text
* Classifying reviews
* Summarizing short paragraphs
* Basic question answering

a **smaller and simpler model is often sufficient**, and using a larger one may only increase cost and latency without improving results.

### 2. Speed

Another important factor is **response speed**.

Larger and more capable models often take longer to generate responses, especially when producing longer outputs. If an application requires real-time interaction, speed becomes critical.

Examples of speed-sensitive applications include:

* Autocomplete systems
* Quick summaries
* Chat interfaces with short answers
* Real-time assistants

In these cases, **faster models are usually the better choice**.

### 3. Input and Output Types

Different applications require different types of inputs and outputs.

Text is the most common format, but many modern AI systems are **multimodal**, meaning they can work with multiple types of data such as:

* Text
* Images
* Audio
* Video

For example:

* An image analysis tool requires **image input support**
* A voice assistant needs **audio input and output**
* A design tool may need **image generation**

If the application requires non-text data, it is important to choose a **model that supports those modalities**.

### 4. Cost

Cost is another key consideration.

Most AI platforms charge based on **tokens**, which represent pieces of text processed by the model. This includes both the input and the generated output.

If an application processes:

* Large documents
* Long conversations
* Large volumes of requests

costs can increase quickly. For this reason, it is important to **compare pricing and choose a model that fits the workload**, rather than automatically selecting the most advanced model.

### 5. Context Window

The **context window** determines how much text a model can process in a single request. This includes:

* The input prompt
* The model's response
* Conversation history (for chat systems)

Applications that require large context windows include:

* Summarizing long documents
* Analyzing large codebases
* Maintaining long conversations

If the context window is too small, the model may lose important information. Therefore, these use cases require models with **larger context capacities**.

### 6. Privacy and Data Control

Privacy can also be an important factor when selecting a model.

If an application handles sensitive information—such as:

* Medical records
* Financial data
* Private company information

sending that data to third-party servers may not be acceptable.

In such cases, **open-source or self-hosted models** may be preferable because they allow organizations to keep data within their own infrastructure.

### 7. Additional Technical Factors

When comparing models, several additional technical characteristics can also influence the decision:

* **Reasoning capability** – how well the model handles complex logic
* **Multimodal support** – whether it accepts images, audio, or video
* **Context window size** – maximum amount of text processed in one request
* **Maximum output tokens** – how long a generated response can be
* **Knowledge cutoff** – the latest date included in the model's training data

Different models may excel in different areas, so these details should be evaluated based on the application's needs.

### Conclusion

There is **no universally best AI model**. The right choice depends on the specific requirements of the application.

When selecting a model, the most important factors to consider are:

* Reasoning ability
* Speed
* Input and output types
* Cost
* Context window size
* Privacy requirements
* Maximum output length
* Knowledge cutoff

The best model is not necessarily the most powerful one—it is the **one that best fits the needs of the application**.

## Understanding model settings

In this lesson, I look at the settings that control how a language model behaves.

On the OpenAI [models page](https://platform.openai.com/chat/edit?models=gpt-4.1), I can inspect a model such as GPT-4.1 and review its key characteristics: intelligence, speed, token pricing, supported modalities, context window, max output tokens, and knowledge cutoff. From there, I can open the model in the Playground and experiment with these settings directly.

Before using the Playground for the first time, I need to add credit to my account. I do that from the billing section in the OpenAI platform. Once I have credit, I can go back to the Playground and start testing prompts with different configurations.

---

### Response format

One of the first settings I can control is the response format.

I have three main options:

* plain text
* JSON object
* JSON schema

#### Plain text

If I choose plain text and send a prompt like:

```text
Give me three benefits of exercising.
```

the model returns a normal text answer. This is the default mode and the most familiar one.

#### JSON object

If I switch the response format to JSON object and ask:

```text
Give me three benefits of exercising as a JSON object.
```

the model returns structured JSON instead of normal prose. That is useful when I want my backend or application code to parse the response reliably.

A possible response could look like this:

```json
{
  "benefits_of_exercising": [
    "Improves cardiovascular and muscular health",
    "Boosts mental well-being and reduces stress",
    "Increases energy and overall stamina"
  ]
}
```

This is already better than plain text when I want to consume the response programmatically.

#### JSON schema

If I need a very specific JSON structure, then a plain JSON object is not enough. In that case, I use JSON schema.

With JSON schema, I define the exact shape of the response I want. For example, instead of getting a loose object with a single array, I may want this structure:

```json
{
  "exercise": {
    "benefits": [
      "Improves cardiovascular and muscular health",
      "Boosts mental well-being and reduces stress",
      "Increases energy and overall stamina"
    ]
  }
}
```

To enforce that shape, I define a schema in the Playground. The lesson shows generating this schema with AI and then using it in the response format settings.

A schema like the one shown in the lesson looks like this:

```json
{
  "name": "exercise_schema",
  "schema": {
    "type": "object",
    "properties": {
      "exercise": {
        "type": "object",
        "properties": {
          "benefits": {
            "type": "array",
            "description": "A list of benefits associated with the exercise.",
            "items": {
              "type": "string",
              "description": "Description of a benefit."
            }
          }
        },
        "required": ["benefits"],
        "additionalProperties": false
      }
    },
    "required": ["exercise"],
    "additionalProperties": false
  }
}
```

Once I save that schema and resend the prompt, the model returns data that follows this exact structure.

This is extremely useful when I need predictable output that must match my application contract.

---

### Temperature

Temperature controls how random or creative the model’s response is.

This setting is usually a value between `0` and `2`, but in practice I should avoid extreme values.

The guideline shown in the lesson is:

* use `0.2 - 0.4` for logical, precise answers
* use `0.7 - 1.0` for creative, expressive tasks
* do not use higher values

So if I am doing summarization, factual answers, extraction, or classification, I should keep temperature low. If I am doing storytelling, brainstorming, or marketing copy, I can raise it.

A good general-purpose default is:

```text
temperature = 0.7
```

That gives a reasonable balance between precision and creativity.

#### What happens with very high temperature

If I push temperature too high, for example to `2`, the model may become too random. The lesson shows an example where the model starts writing a story about a robot, but then the output becomes nonsensical and turns into gibberish.

So the practical lesson is simple: temperature is useful, but extreme values can destroy output quality.

---

### Max tokens

Max tokens controls the maximum length of the response.

If I set it too high, I may spend more tokens than necessary. If I set it too low, the answer may be cut off in the middle of a sentence.

For example, if I ask:

```text
Write a story about a robot.
```

and I set:

```text
max tokens = 15
```

the response may start correctly but get truncated before it finishes.

That means max tokens should always match the type of output I want. If I want a short answer, I should keep it low. If I want a longer explanation or story, I need to allow enough room.

#### A better prompt when max tokens is limited

The lesson also shows an important prompt engineering idea: when I use a lower max token value, I should guide the model more precisely.

Instead of saying:

```text
Write a story about a robot.
```

I can say:

```text
Write a story about a robot in 50 tokens or less. Write the complete answer without cutting off mid-sentence.
```

This gives the model a much better chance of producing a complete response inside the allowed token budget.

So max tokens is not just a model setting. It often works together with prompt design.

---

### Top P

Top P is another way to control randomness.

It works differently from temperature. Instead of directly scaling randomness, it limits the pool of candidate next tokens the model can choose from.

The lesson explains this visually with an example where the current token is followed by several possible next tokens with different probabilities:

* `the` with probability `0.4`
* `a` with probability `0.3`
* `an` with probability `0.1`
* `this` with probability `0.08`
* `one` with probability `0.05`

If I set:

```text
top_p = 1
```

the model can consider the full probability distribution.

If I lower top P, the model focuses only on the most likely next options and ignores the less likely ones.

So top P is also a randomness control, but it works differently from temperature.

The practical rule given in the lesson is:

* change either temperature or top P
* do not try to tune both aggressively at the same time
* if unsure, keep `top_p = 1.0` and tune temperature instead

That is a very good default strategy.

---

### Store logs

Another setting shown in the lesson is store logs.

If this is enabled, prompts and responses are stored on [OpenAI’s platform for logging and debugging](https://platform.openai.com/logs). The lesson shows that I can later go into the dashboard and inspect the logs for each request.

In the logs view, I can see:

* the prompt input
* the model output
* the model used
* the date and time
* token usage
* response format
* max output tokens
* temperature
* a unique response ID

This is useful for debugging, auditing, and understanding exactly how a prompt behaved.

For example, the screenshots show a logged request with details such as:

* a specific prompt about writing a robot story
* the model used
* input and output token counts
* total token count
* text response format
* max output tokens set to `50`
* temperature set to `0.70`

This kind of logging is useful when I need to reproduce results, inspect costs, or debug unexpected behavior.

If I am working with sensitive data, this logging setting becomes important from a privacy perspective, because storing prompts on external servers may not always be appropriate.


>There is no single perfect configuration for every task.

>If I want structured backend-friendly output, I should use JSON or JSON schema. If I want stable, factual responses, I should keep temperature low. If I want creative content, I can raise it moderately. If I want short answers, I should lower max tokens and make the prompt explicit. If I am unsure about top P, I should leave it at `1.0` and focus on temperature instead.

>So the real skill is not just calling the model. It is knowing how to configure these settings so the model behaves the way my application needs.

In this request:

* `temperature` controls randomness

### Enabling streaming

Now I want to make the response appear progressively, token by token, instead of waiting for the whole response to finish.

To do that, I enable streaming by setting:

```js id="nvg3o5"
stream: true
```

I also increase the max output tokens so the streaming effect is easier to observe.

```js id="tbt6u9"
import OpenAI from "openai";

const OPENAI_API_KEY = "your-secret-key";

const client = new OpenAI({
  apiKey: OPENAI_API_KEY,
});

const stream = await client.responses.create({
  model: "gpt-4.1",
  input: "Write a story about a robot.",
  temperature: 0.7,
  max_output_tokens: 250,
  stream: true,
});

for await (const event of stream) {
  console.log(event);
}
```

### Why the variable is now called `stream`

When `stream` is set to `true`, I no longer get a normal response object. Instead, I get a **stream object**.

That stream is an **async iterable**.

This means:

* it is iterable, so I can loop through it
* it is asynchronous, so the values do not all arrive at once
* I receive events one by one as the model generates output in real time

That is why I use:

```js id="42o6fo"
for await (const event of stream) {
  console.log(event);
}
```

The `await` is required because new events arrive progressively while the model is generating output.

### What streamed events look like

When I print the raw events, I see many objects coming through the stream.

The screenshot shows events like this:

```js id="576dtw"
{
  type: "response.output_text.delta",
  sequence_number: 252,
  item_id: "msg_...",
  output_index: 0,
  content_index: 0,
  delta: "a",
  logprobs: []
}
```

and similar ones with increasing sequence numbers such as `253`, `254`, and so on.

The most important field here is:

```js id="78fkrb"
event.delta
```

That contains the chunk of text currently being generated.

The `type` field tells me what kind of event it is. In this case:

```js id="yfniat"
type: "response.output_text.delta"
```

means this event is a text chunk generated during streaming.

### Printing only the streamed text chunks

Instead of logging the entire event object, I can print only the `delta` field:

```js id="o2vbyf"
import OpenAI from "openai";

const OPENAI_API_KEY = "your-secret-key";

const client = new OpenAI({
  apiKey: OPENAI_API_KEY,
});

const stream = await client.responses.create({
  model: "gpt-4.1",
  input: "Write a story about a robot.",
  temperature: 0.7,
  max_output_tokens: 250,
  stream: true,
});

for await (const event of stream) {
  console.log(event.delta);
}
```

This gets me much closer to the ChatGPT-style typing effect.

### Why `undefined` appears

If I print `event.delta` directly for every event, I may see a bunch of `undefined` values at the beginning and end.

That happens because **not every event contains a `delta` property**.

Some events represent:

* the start of the response
* metadata
* the end of the response
* other internal streaming events

Only the events that actually contain text chunks have `delta`.

So I need to guard against that.

### Proper streaming output

The correct approach is to print `event.delta` only if it exists:

```js id="53737t"
import OpenAI from "openai";

const OPENAI_API_KEY = "your-secret-key";

const client = new OpenAI({
  apiKey: OPENAI_API_KEY,
});

const stream = await client.responses.create({
  model: "gpt-4.1",
  input: "Write a story about a robot.",
  temperature: 0.7,
  max_output_tokens: 250,
  stream: true,
});

for await (const event of stream) {
  if (event.delta) {
    console.log(event.delta);
  }
}
```

This removes the `undefined` messages.

### Printing on one line instead of many lines

There is still one problem: `console.log()` adds a newline after every chunk, so each token or fragment appears on a separate line.

To make the output appear continuously on the same line, like a real streaming chat experience, I should use `process.stdout.write()` instead.

```js id="tmqrr3"
import OpenAI from "openai";

const OPENAI_API_KEY = "your-secret-key";

const client = new OpenAI({
  apiKey: OPENAI_API_KEY,
});

const stream = await client.responses.create({
  model: "gpt-4.1",
  input: "Write a story about a robot.",
  temperature: 0.7,
  max_output_tokens: 250,
  stream: true,
});

for await (const event of stream) {
  if (event.delta) {
    process.stdout.write(event.delta);
  }
}
```

This produces the same kind of progressive text rendering I see in ChatGPT.
